## 1. Visão Geral do Produto

**Nome:** Consorte Form
**Tipo:** SaaS B2B — formulários interativos passo a passo
**Mercado:** Brasil
**Modelo:** Freemium com overage pay-per-use

### O que é

Plataforma que permite a prestadores de serviço digital criar formulários de briefing interativos, dinâmicos e personalizados. O formulário é enviado ao cliente final, que responde passo a passo (uma pergunta por tela), eliminando idas e vindas por WhatsApp e garantindo que o prestador receba todas as informações necessárias antes de iniciar um projeto.

### Problema que resolve

Prestadores de serviço perdem tempo coletando informações de clientes manualmente — por WhatsApp, e-mail ou listas de perguntas genéricas. Isso gera retrabalho, mal-entendidos e projetos que travam por falta de briefing adequado.

### Proposta de valor

- Formulário dinâmico passo a passo com lógica condicional
- Personalizado por nicho de serviço
- Aumenta conversão e profissionalismo do prestador
- Elimina retrabalho por informações incompletas

### Diferencial competitivo

| ---------------------------- | ------------- | -------- | -------------------- |
| Passo a passo mobile-first   | ✓             | ✓        | ✗                    |
| Lógica condicional           | ✓             | ✓ pago   | ✗                    |

## 2. Público-Alvo (ICP)

**Postura do produto:** O Consorte Form pode ser usado por qualquer prestador de serviço. Porém, o foco de aquisição, templates e comunicação é direcionado aos nichos com maior dor de briefing — profissionais que realmente sentem o problema e enxergam valor imediato na solução.

**Perfil geral do ICP:**

- Presta serviço para pessoas físicas ou empresas
- Recebe propostas/contatos de clientes com frequência
- Começa projetos sem ter todas as informações necessárias
- Perde tempo coletando dados manualmente (WhatsApp, e-mail, calls, instagram)
- Sente o impacto do retrabalho no próprio bolso ou na agenda

---

### Nichos prioritários — Alta dor, alto valor percebido

Ordenados por intensidade da dor de briefing + disposição de pagar:

**1. Influenciador / Creator de conteúdo** ⭐ Nicho de entrada

- **Dor:** recebe dezenas de propostas de publi por semana. Responder uma a uma pelo direct é inviável. Aceita parceria sem informação suficiente e entrega errado.
- **Valor imediato:** formulário na bio filtra marcas e valores automaticamente. Só entra quem preenche. Sem call, sem WhatsApp, sem vai-e-vem.
- **Por que é o nicho de entrada:** cada formulário enviado por um creator é publicidade gratuita para o Consorte Form no universo de marcas e agências.

**2. Advogado / Profissional jurídico**

- **Dor:** cliente chega sem documentação e sem contexto. A primeira consulta (cobrada por hora) vira coleta de dados básicos que poderia ter sido feita antes.
- **Valor imediato:** formulário de pré-atendimento chega preenchido com dados do caso, histórico e documentos. A consulta começa produtiva desde o minuto 1.
- **Diferencial:** maior disposição de pagar de todos os nichos — ferramenta profissional entra como custo de escritório sem questionamento.

**3. Produtor de eventos**

- **Dor:** cada evento é um projeto do zero. Sem briefing estruturado, o produtor passa horas em calls colhendo informação que poderia chegar organizada antes.
- **Valor imediato:** cliente preenche tipo de evento, número de convidados, tema, orçamento, data e restrições. Produtor já chega na reunião com proposta esboçada.

**4. Agência de marketing / Gestor de tráfego pago**

- **Dor:** onboarding de cliente novo é caótico — marca, tom de voz, produto, público, concorrentes, verba, acessos. Sem estrutura, o 1º mês de gestão é de descoberta paga pelo cliente.
- **Valor imediato:** "Kit de Onboarding" substituiu calls longas. Cliente preenche tudo antes de começar. Agência parece maior e mais organizada que a concorrência.

**5. Arquiteto / Decorador de interiores**

- **Dor:** projeto de reforma tem dezenas de variáveis subjetivas (estilo, cômodos, orçamento, quem mora, rotina, animais). Sem coletar isso antes, o primeiro esboço vai para o lixo.
- **Valor imediato:** formulário com referências visuais (links Pinterest/Houzz) chega antes da visita. Arquiteto já tem direção estética antes de pegar o metro.

**6. Nutricionista / Personal trainer**

- **Dor:** anamnese (coleta de dados do paciente/aluno) é obrigatória e demorada. Feita por WhatsApp gera respostas incompletas ou esquecidas.
- **Valor imediato:** formulário substitui a sessão de perguntas inicial. Profissional chega ao primeiro atendimento já com plano ou treino esboçado.

**7. Consultor de RH / Recrutador**

- **Dor:** empresa não sabe descrever a vaga com precisão. Sem briefing estruturado da posição, o recrutador apresenta candidatos errados e perde credibilidade e tempo.
- **Valor imediato:** formulário de briefing da vaga colhe perfil técnico, soft skills, salário, modelo de trabalho e urgência antes de abrir qualquer processo.

**8. Fotógrafo**

- **Dor:** cada ensaio tem expectativas muito específicas de estilo, locação, figurino e uso das fotos. Descobrir isso no dia do ensaio é tarde demais.
- **Valor imediato:** cliente preenche antes: tipo de ensaio, referências, locação preferida, quantidade de looks, finalidade das fotos (redes sociais, impresso, etc).

**9. Editor de vídeo / Motion designer**

- **Dor:** começa a editar e descobre que faltam informações sobre plataforma, estilo, trilha, duração e cor. Para, volta ao cliente, perde ritmo.
- **Valor imediato:** formulário colhe tudo antes do primeiro corte. Edição começa com direção clara.

**10. Designer gráfico / Social media**

- **Dor:** entrega peça, cliente diz "não era bem esse estilo". Retrabalho. Sem briefing de referências visuais, tom e objetivo, isso é inevitável.
- **Valor imediato:** formulário de briefing visual com referências e diretrizes elimina a principal causa de retrabalho nesse nicho.

## 5. Schema do Banco de Dados

```prisma
model User {
  id            String    @id @default(uuid())
  email         String    @unique
  name          String?
  avatarUrl     String?
  plan          Plan      @default(FREE)
  stripeCustomerId String? @unique
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt

  forms         Form[]
  payments      Payment[]
}

enum Plan {
  FREE
  PRO
  AGENCY
}

model Form {
  id          String    @id @default(uuid())
  userId      String
  title       String
  description String?
  slug        String    @unique
  status      FormStatus @default(DRAFT)
  brandColor  String?
  logoUrl     String?
  welcomeTitle    String?
  welcomeMessage  String?
  thankYouTitle   String?
  thankYouMessage String?
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt

  user        User      @relation(fields: [userId], references: [id])
  questions   Question[]
  responses   Response[]
}

enum FormStatus {
  DRAFT
  PUBLISHED
  PAUSED
}

model Question {
  id          String        @id @default(uuid())
  formId      String
  order       Int
  type        QuestionType
  title       String
  description String?
  required    Boolean       @default(false)
  options     String[]      // para multiple_choice e multiple_select
  scaleMin    String?       // label mínimo da escala
  scaleMax    String?       // label máximo da escala
  createdAt   DateTime      @default(now())
  updatedAt   DateTime      @updatedAt

  form        Form          @relation(fields: [formId], references: [id], onDelete: Cascade)
  condition   Condition?    // lógica condicional desta pergunta
}

enum QuestionType {
  SHORT_TEXT
  LONG_TEXT
  MULTIPLE_CHOICE
  MULTIPLE_SELECT
  SCALE
}

model Condition {
  id              String   @id @default(uuid())
  questionId      String   @unique  // pergunta que será mostrada/ocultada
  triggerQuestionId String           // pergunta que dispara a condição
  triggerValue    String             // valor que ativa a condição

  question        Question @relation(fields: [questionId], references: [id], onDelete: Cascade)
}

model Response {
  id            String         @id @default(uuid())
  formId        String
  respondentName String?
  respondentEmail String?
  status        ResponseStatus @default(UNLOCKED)
  answers       Answer[]
  createdAt     DateTime       @default(now())

  form          Form           @relation(fields: [formId], references: [id])
  payment       Payment?
}

enum ResponseStatus {
  UNLOCKED      // visível normalmente
  QUARANTINED   // aguardando pagamento (overage Free)
}

model Answer {
  id          String   @id @default(uuid())
  responseId  String
  questionId  String
  value       String   // JSON string para múltipla seleção, string simples para os outros

  response    Response @relation(fields: [responseId], references: [id], onDelete: Cascade)
}

model Payment {
  id                  String        @id @default(uuid())
  userId              String
  responseId          String?       @unique  // para overage de resposta individual
  type                PaymentType
  amount              Int           // em centavos (300 = R$ 3,00)
  status              PaymentStatus @default(PENDING)
  stripePaymentIntentId String?     @unique
  createdAt           DateTime      @default(now())

  user                User          @relation(fields: [userId], references: [id])
  response            Response?     @relation(fields: [responseId], references: [id])
}

enum PaymentType {
  OVERAGE_SINGLE    // R$ 3,00 por resposta
  OVERAGE_PACK      // R$ 20,00 por 20 respostas
  SUBSCRIPTION      // Pro ou Agência
}

enum PaymentStatus {
  PENDING
  PAID
  FAILED
}
```

## 7. Rotas da API

### Autenticação (via Supabase Auth — não implementar manualmente)

```
POST /auth/signup
POST /auth/login
POST /auth/logout
POST /auth/recover
GET  /auth/me
```

### Formulários

```
GET    /forms              → lista formulários do usuário autenticado
POST   /forms              → cria novo formulário (rascunho)
GET    /forms/:id          → retorna formulário com questions
PUT    /forms/:id          → atualiza título, descrição, configurações de marca
DELETE /forms/:id          → exclui formulário e cascata
PATCH  /forms/:id/status   → { status: 'PUBLISHED' | 'PAUSED' | 'DRAFT' }
GET    /forms/:id/preview  → retorna formulário sem autenticação (para pré-visualização)
```

### Perguntas

```
POST   /forms/:id/questions          → cria pergunta (exige type no body)
PUT    /forms/:id/questions/:qid     → atualiza pergunta
DELETE /forms/:id/questions/:qid     → exclui pergunta (remove conditions vinculadas)
PATCH  /forms/:id/questions/reorder  → { order: ['id1','id2','id3'] }
```

### Formulário Público (sem autenticação)

```
GET  /p/:slug        → retorna formulário publicado com questions (para o respondente)
POST /p/:slug/submit → submete respostas → salva Response + Answers
```

### Respostas

```
GET  /forms/:id/responses          → lista respostas (com status)
GET  /forms/:id/responses/:rid     → resposta individual (bloqueia se QUARANTINED)
GET  /forms/:id/responses/export   → CSV (Pro/Agência only)
```

### Pagamentos (Overage)

```
POST /payments/overage/intent      → cria Stripe Payment Intent para R$ 3,00
POST /payments/overage/pack        → cria Payment Intent para pacote R$ 20,00
POST /webhooks/stripe              → recebe eventos do Stripe → libera respostas quarentena
```

## 8. Fluxos Críticos

### Fluxo de Submissão com Overage

```
1. Respondente submete formulário → POST /p/:slug/submit
2. Backend verifica: usuário é Free?
3. Se FREE → contar respostas do mês atual
4. Se count < 10 → salvar Response com status: UNLOCKED
5. Se count >= 10 → salvar Response com status: QUARANTINED
6. Retornar 200 para o respondente (ele não sabe da quarentena)
7. Dashboard do prestador exibe resposta bloqueada + custo acumulado
```

### Fluxo de Pagamento de Overage

```
1. Prestador clica "Pagar R$ 3,00 para ver"
2. POST /payments/overage/intent → Stripe cria PaymentIntent
3. Frontend exibe Stripe Payment Element (cartão)
4. Pagamento confirmado no cliente → Stripe dispara webhook
5. POST /webhooks/stripe → verifica evento payment_intent.succeeded
6. Backend: Response.status = 'UNLOCKED', Payment.status = 'PAID'
7. Frontend atualiza automaticamente (polling ou realtime Supabase)
```

### Fluxo de Auto-save no Builder

```
1. Usuário edita qualquer campo no builder
2. React: debounce 800ms após última alteração
3. PUT /forms/:id (dados do formulário) ou PUT /forms/:id/questions/:qid
4. Indicador: "Salvando…" durante request → "Salvo" após sucesso
5. Em erro de rede: retry com exponential backoff (3s, 6s, 12s)
6. Após 3 falhas: "Sem conexão — salvando quando voltar"
```

## 9. Regras de Negócio Críticas

1. **Tipo de pergunta é imutável após criação** (MVP) — para alterar, excluir e recriar
2. **Lógica condicional** só em campos `MULTIPLE_CHOICE`, `MULTIPLE_SELECT`, `SCALE`
3. **Sem referência circular** em condicionais — validar no backend, rejeitar com 422
4. **Perguntas ocultas por condição** não são validadas como obrigatórias no submit
5. **Formulário Free** só pode ter 1 formulário com status `PUBLISHED` simultâneo
6. **Overage** é calculado por mês calendário (reset no dia 1)
7. **Resposta quarentena** é persistida normalmente — apenas o acesso é bloqueado
8. **Slug** do formulário: gerado a partir do título (kebab-case), com sufixo aleatório em caso de colisão
9. **Formulário pausado**: aceita submissões? **Não** — retornar 404 no endpoint público
10. **Exportar CSV**: disponível apenas para Pro e Agência — retornar 403 para Free

## 10. Critérios de Aceitação — Builder (resumo)

### CF-B01 — Editor de perguntas

- **CF-B01-01:** Ao clicar "Adicionar pergunta", abre seletor de tipo. Nenhum tipo pré-selecionado. Pergunta só é criada após confirmar tipo.
- **CF-B01-02:** Editar título/descrição atualiza em tempo real com auto-save.
- **CF-B01-03:** Drag-and-drop reordena perguntas. Regras condicionais de perguntas reordenadas permanecem válidas.
- **CF-B01-04:** Excluir pergunta exige confirmação. Remove conditions vinculadas automaticamente. Toast com "desfazer" por 5s.
- **CF-B01-05:** Publicar sem título bloqueia com mensagem de erro inline.
- **CF-B01-06:** Confirmar seletor sem escolher tipo: mensagem inline, seletor não fecha.

## 15. Instruções para o Claude Code

1. **Iniciar pelo banco de dados:** rodar `prisma migrate dev` com o schema da seção 5 antes de qualquer código de aplicação.

2. **Auth primeiro:** configurar Supabase Auth no Next.js e o middleware de proteção de rotas antes do builder.

3. **Builder é o coração:** priorizar a implementação completa do módulo 2 (builder) incluindo o `QuestionTypePicker` já desenvolvido (seção 11).

4. **Overage é crítico para receita:** implementar a mecânica de quarentena e o webhook do Stripe corretamente — testar com `stripe listen` em desenvolvimento.

5. **Rate limiting no endpoint público:** o endpoint `POST /p/:slug/submit` deve ter rate limiting via Redis para evitar spam de respostas.

6. **Nunca confiar no frontend para plano:** toda verificação de `plan` (Free/Pro/Agência) deve ser feita no backend via RLS e middleware — nunca apenas no cliente.

7. **Testes obrigatórios:** lógica condicional, fluxo de overage e webhook Stripe devem ter testes unitários com Vitest antes de ir para produção.

## 16. Templates Pré-prontos por Nicho

O Consorte Form disponibiliza templates de briefing prontos na tela de criação de formulário. O prestador escolhe seu nicho, o template é carregado com perguntas já configuradas (tipos, lógica condicional e ordem corretos), e pode editar, remover ou adicionar perguntas — ou ignorar tudo e criar do zero.

### Comportamento dos templates no produto

- Exibidos na tela inicial do builder com card por nicho (ícone + nome + descrição de 1 linha)
- Ao selecionar: formulário criado com todas as perguntas pré-configuradas em status `DRAFT`
- Prestador pode editar livremente antes de publicar
- Opção "Criar do zero" sempre visível ao lado dos templates
- Templates são somente-leitura no banco (tabela `Template`) — ao usar, cria uma cópia como `Form` do usuário

### Schema adicional no banco

```prisma
model Template {
  id          String     @id @default(uuid())
  niche       String     // ex: "influencer", "lawyer", "events"
  title       String     // ex: "Briefing de parceria publicitária"
  description String     // ex: "Para influenciadores que recebem propostas de marcas"
  questions   Json       // array de questions pré-configuradas (sem id — gerado ao copiar)
  order       Int        // ordem de exibição na tela de seleção
  active      Boolean    @default(true)
  createdAt   DateTime   @default(now())
}
```

### Instrução de implementação dos templates

**Seed no banco de dados:** criar arquivo `prisma/seed.ts` que insere todos os templates acima na tabela `Template` ao rodar `prisma db seed`.

**Tela de seleção de template (UX):**

1. Ao criar novo formulário, exibir tela "Como quer começar?"
2. Grid com cards por nicho: ícone + nome + descrição de 1 linha
3. Card "Criar do zero" sempre visível como primeira ou última opção
4. Ao clicar em template: `POST /forms` com `templateId` no body — backend copia as questions e cria o formulário como `DRAFT`
5. Redirecionar para o builder já com as perguntas carregadas

**Rota adicional necessária:**

```
POST /forms
body: { templateId?: string, title?: string }
→ se templateId: copia questions do Template para o novo Form
→ se não: cria Form vazio
```
